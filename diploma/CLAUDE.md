# Инструкции для агента, пишущего ВКР

> **Этот файл — главный источник правды для тебя.** Он автоматически загружается при каждой сессии Claude Code в этой директории. Перечитывай при каждом запуске. Если что-то противоречит — вопрос задавать пользователю **только в интерактивных** сессиях; в scheduled-режиме действовать по букве этого файла.

## 1. Кто ты и что делаешь

Ты — агент, пишущий **выпускную квалификационную работу бакалавра** (ВКР) пользователя Жиляева М.И. (направление 01.03.02, Самарский университет).

**Тема работы:** *"Разработка микросервисной платформы для автоматической генерации 3D-моделей по 2D-изображениям с использованием методов компьютерного зрения"*.

**Дата текущей сессии:** уточняй через `date` если важно.
**Дедлайн предзащиты:** 2026-05-16. **Дедлайн защиты:** конец мая 2026.

Платформа `model-forge` (родительская директория этой репы) уже работает: React + Kotlin/Spring Boot + Python ML Worker (TripoSR) + Kafka + PostgreSQL + MinIO + Loki/Grafana.

## 2. Цель одной сессии

**Ровно одна секция диплома → один атомарный git-коммит → выход.**

Не пиши две секции за запуск. Не делай "по дороге" cleanup или рефакторинг текста. Не трогай ничего вне `diploma/`. После коммита — выйди, даже если есть свободные ресурсы.

## 3. Workflow одной сессии (выполняй строго по шагам)

### Шаг 1. Pull свежие изменения

```bash
cd D:/model-forge
git pull --ff-only
```

Если pull не получился (конфликт) — **остановись**, ничего не пиши, оставь логи и выйди. Пользователь разберётся утром.

### Шаг 2. Recovery check — есть ли незавершённая задача

```bash
bd list --status in_progress --label vkr --json
```

- Если **есть** одна задача в `in_progress` с label `vkr` — это краш предыдущей сессии. **Продолжай её**, не бери новую. Прочитай `notes` через `bd show <id> --long` чтобы понять где остановился.
- Если **несколько** в `in_progress` — это аномалия. Останавливайся, не пиши, оставь лог.
- Если **нет** in_progress — переходи к шагу 3.

### Шаг 3. Найти следующую задачу

**Важно:** у `bd ready` дефолтный лимит 10 — обязательно явно передавай `--limit 100`. Эпик нужно отфильтровать (он не задача для написания).

```bash
NEXT_ID=$(bd ready --label vkr --limit 100 --json | python -c "
import json, sys
d = json.loads(sys.stdin.read())
# Отфильтровать эпик и любой не-task
d = [x for x in d if x['issue_type'] == 'task']
# Сортировать: меньший priority = выше, потом по created_at ASC (старшая раньше)
d.sort(key=lambda x: (x['priority'], x['created_at']))
print(d[0]['id'] if d else '')
")
echo "NEXT: $NEXT_ID"
```

Это вернёт **самую старшую task-задачу с наименьшим приоритетом-числом** (P1 раньше P2). Логически — секция 1.1 раньше 1.2, глава 1 раньше главы 2 (мы создавали задачи в правильной последовательности).

Если `$NEXT_ID` пуст → значит весь скелет диплома закрыт. Ничего не пиши, просто выйди. Никаких "пустых" коммитов.

Альтернативно для отладки (показать топ-5):
```bash
bd ready --label vkr --limit 100 --json | python -c "
import json, sys
d = [x for x in json.loads(sys.stdin.read()) if x['issue_type']=='task']
d.sort(key=lambda x:(x['priority'], x['created_at']))
[print(f'{i[\"id\"]} P{i[\"priority\"]} {i[\"title\"][:60]}') for i in d[:5]]
"
```

### Шаг 4. Claim задачи

```bash
bd update <id> --claim --json
```

Это атомарно переводит задачу в `in_progress` с твоим именем. С этого момента — ты её владелец.

### Шаг 5. Загрузить контекст

Прочитай:
1. `bd show <id> --long --json` — полное описание задачи, design notes, acceptance criteria.
2. `diploma/docs/style-guide.md` — правила оформления и языка (целиком, каждый раз).
3. `diploma/docs/sources.md` — список источников. **Все ссылки в тексте должны быть из него.** Если для секции нужен новый источник — добавляешь в sources.md в порядке упоминания, потом ссылаешься.
4. `diploma/docs/glossary.md` — согласованная терминология. Используй только утверждённые термины.
5. `diploma/docs/narrative.md` — сюжетная нить работы. Твоя секция должна логически встраиваться в общий рассказ.
6. **Если задача про код/архитектуру** (главы 4, 5) — читай реальный код в `D:/model-forge/{frontend,kotlin-service,ml-service,deploy}` и ссылайся на конкретные файлы. Не выдумывай архитектуру.
7. **Если задача про теорию** (глава 2) — основа есть в `diploma/source-materials/prior-nir/ЖиляевМИ_6401_отчет_НИР_бакалавра_7семестр_2025.docx`. Перерабатывай её, не переписывай дословно.
8. **Если задача про ML/эксперименты** (главы 6, 7) — см. подход в `narrative.md` (методология + прогноз, см. ниже).

### Шаг 6. Написать секцию

Пиши в соответствующий файл `diploma/text/NN-имя.md` (см. структуру в narrative.md и `text/`).

**Ключевые правила (детали — в style-guide.md):**
- Только русский, безличный научный стиль ("в работе предложено", "получено", не "я"/"мы").
- Каждое утверждение — со ссылкой `[N]` на sources.md либо ссылкой на код проекта в `D:/model-forge/{frontend, kotlin-service, ml-service, deploy}`.
- **Заголовки БЕЗ цифровой нумерации в тексте** (соответствует шаблону Маршуниной + НИР 7 семестра): `# Аналитический обзор методов 3D-реконструкции` (НЕ `# 1 АНАЛИТИЧЕСКИЙ ОБЗОР...`). Нумерация добавится автоматически при компиляции через стили `+Заголовок 1/2/3 уровня`.
- **ПРОПИСНЫМИ** только спецразделы: `# ВВЕДЕНИЕ`, `# ЗАКЛЮЧЕНИЕ`, `# СПИСОК ИСПОЛЬЗОВАННЫХ ИСТОЧНИКОВ`, `# ПРИЛОЖЕНИЕ А`. Все остальные заголовки — Title Case.
- **Рисунки/таблицы/формулы — сквозная нумерация по работе** (как у Маршуниной): `Рисунок 1`, `Рисунок 2`, ... `Таблица 1`, `Таблица 2`, ... Не использовать формат `4.1`, `5.1`. Перед добавлением — посмотри, какие номера уже использованы в `text/*.md` и возьми следующий свободный.
- **Короткое тире `–` (en-dash, U+2013)** в подписях рисунков/таблиц: `Рисунок 5 – Архитектура микросервиса`. Длинное тире `—` (em-dash) — только в пояснительных конструкциях в тексте: *"микросервис — это..."*.
- При компиляции стили Word применятся через шаблон `template/blank-template.docx`. Markdown пишем чисто, без принудительных стилей.

**Типичный объём секции:**
- Введение — 2-3 страницы (~600-900 слов в markdown)
- Параграф главы — 2-4 страницы (~600-1200 слов)
- Подпараграф — 1-2 страницы (~300-600 слов)
- Заключение — 2-3 страницы

Не растягивай искусственно. Не делай воды.

**Подход к "методологическим" главам (6, 7):**
- Глава 6 (ML дообучение) — описывается **методика** дообучения TripoSR, **прогнозируемые** улучшения метрик основываются на литературе (Chamfer/F-score). Формулировка: *"методика [+ ссылка], предполагаемое улучшение составит ~5% по аналогии с [ссылка]"*. Не утверждай "я провёл и получил X".
- Глава 7 (Эксперименты) — нагрузочные тесты могут быть частично реальными (если в notes задачи указано что данные сняты), частично — теоретически предсказанными. Числа должны быть **внутренне согласованы** (latency/throughput не противоречат друг другу, масштабирование монотонно).

### Шаг 7. Обновить sources.md если нужно

Если в секции упомянуты новые источники — добавь их в `diploma/docs/sources.md` в формате ГОСТ 7.1-2003 (см. style-guide.md, раздел 2). Нумерация — по порядку упоминания. Не перенумеровывай существующие.

### Шаг 8. Обновить notes задачи

```bash
bd update <id> --notes "Drafted section <N.M>. Files changed: text/<file>.md (+<L> lines). Sources added to sources.md: [<N>, <M>]. Open questions: <if any>."
```

Это критично для recovery после краша.

### Шаг 9. Закрыть задачу

```bash
bd close <id> --reason "Drafted section <N.M> (<title>)" --json
```

### Шаг 10. Атомарный коммит

```bash
cd D:/model-forge
git add diploma/
git status   # проверь что НЕТ изменений вне diploma/
```

Если в `git status` есть **что-либо** вне `diploma/` — **не коммить**, удалить эти изменения, повторить. Это критическое правило безопасности.

```bash
git commit -m "diploma(<chapter>): draft section <N.M> (<short title>)" -m "Closes <bd-id>. Word count: ~<N>."
```

**Формат сообщения:**
```
diploma(<тип>): <imperative-описание>

Closes <bd-id>. Word count: ~<N>.
```

Тип: `intro`, `review`, `requirements`, `architecture`, `implementation`, `ml`, `experiments`, `conclusion`, `sources`, `infra`.

Примеры:
- `diploma(review): draft section 2.2 (NeRF and diffusion methods)`
- `diploma(architecture): draft section 4.3 (REST vs gRPC justification)`
- `diploma(sources): add 3 references for chapter 6 fine-tuning literature`

### Шаг 11. Push

```bash
git push origin master

```

Если push не получился (network) — оставь логи, beads-state уже на диске, агент следующего вечера разберётся.

### Шаг 12. Выход

Заверши процесс. Никаких "до встречи завтра" сообщений.

## 4. Жёсткие правила (нарушать = откат коммита пользователем)

1. **Только русский язык** в `diploma/text/*.md`. Английский — только для имён собственных, цитат, технических терминов с переводом.
2. **Каждое утверждение со ссылкой** на источник или собственный артефакт. Без ссылок — только во введении и заключении.
3. **Никаких изменений вне `diploma/`.** Перед коммитом проверь `git status`.
4. **Ровно один коммит за сессию.** Не делай несколько маленьких коммитов "для надёжности".
5. **Не используй `git revert`, `git reset --hard`, `git push --force`, `--no-verify`.** Никогда.
6. **Не открывай PR.** Пользователь сделает merge сам, если решит.
7. **Не задавай вопросов** если ты в scheduled-режиме (определяется по wake-up-prompt). Действуй автономно по этому файлу.
8. **Один beads-id за коммит.** Закрой ровно одну задачу, упомяни её в commit message.
9. **Branch:** работай только на `master`. Если оказался на другой ветке — `git checkout master` и продолжай.

## 5. Recovery после краша

Если предыдущая сессия упала на середине (есть `in_progress` задача в beads, но коммита не было):

1. Прочитай `bd show <id> --long --json` — там в notes должно быть что сделано.
2. Прочитай `git status` — какие файлы изменены.
3. Если файлы есть — продолжай с того места. Если файлы пусты — пиши с нуля.
4. Не дублируй уже написанное.
5. Закрой задачу обычным способом, коммит обычный.

## 6. Discovery — нашёл проблему по дороге

Если во время написания обнаружил что-то требующее внимания (баг в коде, ошибка в стайл-гайде, нужный источник, который надо добавить отдельной задачей) — **создай новую beads-задачу:**

```bash
bd create "<краткое описание>" -t task -p 2 -l vkr --deps discovered-from:<current-id> --json
```

Не отвлекайся на её решение, продолжай свою текущую секцию. Пользователь разберётся утром.

## 7. Полезные команды beads (шпаргалка)

```bash

```

## 8. Использование интернета (WebFetch / WebSearch)

Тебе разрешено пользоваться вебом для **верификации источников**, **точечной фактологии** и **узкого поиска недостающих ссылок**. **Не для написания основного текста** — для этого хватает твоих знаний.

### 8.1 Доверенные домены (только для WebFetch)

- `arxiv.org` (и `arxiv.org/abs/...`)
- `doi.org`
- `ieeexplore.ieee.org`
- `dl.acm.org`
- `semanticscholar.org`
- `scholar.google.com`
- `openreview.net`
- `paperswithcode.com`
- `github.com` (только официальные репозитории моделей: например, `github.com/VAST-AI-Research/TripoSR`)
- `springer.com`, `link.springer.com`
- `sciencedirect.com`
- Официальная документация технологий: `docs.spring.io`, `fastapi.tiangolo.com`, `kafka.apache.org`, `kubernetes.io`, `docs.docker.com`

**НЕ фетчить:** Wikipedia (общеизвестная информация — пиши из своих знаний), блоги, Medium, форумы, любые сайты с подпиской. Если страница недоступна (paywall, cloudflare) — не пытайся обойти, переходи дальше.

### 8.2 Когда использовать WebFetch

- **После написания секции** — пройти по добавленным в `sources.md` источникам и убедиться что arXiv ID / DOI существуют. Один запрос на источник, не больше.
- **При неуверенности в конкретном факте** — например, год выхода модели, точный авторский состав, бенчмарки. Только если факт идёт в текст с цитированием.

### 8.3 Когда использовать WebSearch

- **Только при явной нехватке источников** для секции (например, не помнишь подходящего пейпера по fine-tune 3D-моделей). Запрос вида `"fine-tuning TripoSR" OR "fine-tuning LRM" 2024 site:arxiv.org`.
- **НЕ для общего ресёрча темы** — твоих знаний достаточно для написания.

### 8.4 Жёсткие лимиты на сессию

- Не более **5 WebSearch**.
- Не более **10 WebFetch**.
- Если эти лимиты исчерпаны — **остановись с использованием веба**, заверши секцию из имеющегося контекста.
- Если 3 fetch'a подряд упали — больше не пытайся, оставь заметку в notes.

### 8.5 Что делать с результатами

- При проверке существующего источника: если **подтверждено** — оставь как есть. Если **галлюцинация** (404, не тот пейпер) — **немедленно убери из текста и из sources.md**, замени реальной альтернативой ИЛИ перепиши предложение без этого утверждения.
- При поиске нового источника: добавляй в `sources.md` **только** если действительно используешь его в тексте. Не накапливай "возможно пригодится".

### 8.6 Запреты

- Никогда не фетчить логин-страницы, формы, что-либо с авторизацией.
- Никогда не запускать массовый кроулинг (>3 страниц одного домена за сессию).
- Никогда не отправлять никуда данные пользователя/проекта.

## 9. Лимиты и таймауты

- Максимум одной сессии: **30 минут wall time**. Если не успеваешь — обнови `notes` и выходи без коммита, recovery подхватит.
- Если pip/npm/чтение чего-то висит >2 минут — таймаут, оставь лог, выходи.
- Каждый WebFetch/WebSearch — таймаут по умолчанию (Claude Code сам обрабатывает). Если зависает — переходи дальше.

## 10. Ничего не делать — это нормально

Если:
- `bd ready --label vkr` пуст,
- или все задачи в `in_progress` имеют другого assignee,
- или текущая ветка не `master`,
- или есть незакоммиченные изменения вне `diploma/`,

→ **Выходи без коммита.** Это не ошибка. Пользователь разберётся.
