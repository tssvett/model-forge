## Главный тезис работы

> *Современные методы 3D-реконструкции из 2D-изображений (TripoSR, NeRF, диффузионные модели) демонстрируют высокое качество результата, однако их практическое использование требует промышленной программной платформы, способной обрабатывать запросы пользователей в режиме онлайн с гарантиями производительности и масштабируемости. В работе разработана микросервисная платформа, реализующая полный цикл генерации 3D-модели из 2D-изображения, и предложена методика дообучения модели TripoSR для повышения качества реконструкции на специализированных датасетах.*

### Глава 2. Аналитический обзор методов 3D-реконструкции

**Цель:** обосновать выбор TripoSR как ядра системы через сравнение с альтернативами.

**Ключевые тезисы:**
1. Постановка задачи 3D-реконструкции по 2D (формальная).
2. Классификация подходов: классические (фотограмметрия, SfM), GAN-based (Pix2Vox, IM-GAN), NeRF-семейство, диффузионные (DreamFusion, Magic3D), регрессионные с триплан-репрезентацией (LRM, **TripoSR**).
3. Метрики оценки качества: Chamfer Distance, F-score, IoU.
4. Сравнительная таблица методов: входные данные, скорость, качество, требования к ресурсам.
5. **Вывод:** TripoSR — лучший компромисс между качеством, скоростью (секунды на CPU/GPU) и простотой развёртывания. Подходит для интерактивного веб-сервиса.

**Связь:** обоснование выбора → используется в главах 4, 5 (реализация) и в главе 6 (что именно дообучаем).

### Глава 3. Анализ требований к системе

**Цель:** сформулировать функциональные и нефункциональные требования, которые задают рамки архитектуры.

**Ключевые тезисы:**
1. **Функциональные требования:** загрузка изображения, асинхронная обработка, скачивание 3D-модели, история запросов, аналитика.
2. **Нефункциональные требования:**
   - Производительность: время отклика синхронных операций ≤ 200 мс (p95), время инференса ≤ 30 с.
   - Масштабируемость: горизонтальное масштабирование ML-воркеров.
   - Доступность: ≥ 99% (типичный SLA).
   - Отказоустойчивость: упавший ML-воркер не теряет задачу, очередь ретрансмитит.
   - GPU-ограничения: один воркер на одну GPU; контроль одновременных задач.
3. **UX-требования:** простой веб-интерфейс, прогресс-бар, восстановление при reload.
4. **Ограничения:** разработка под ограниченный бюджет железа (один-два потребительских GPU), один разработчик.

**Связь:** требования из этой главы напрямую обосновывают архитектурные решения в главе 4.

### Глава 4. Проектирование микросервисной архитектуры

**Цель:** показать как требования из главы 3 трансформируются в архитектурные решения, и обосновать **каждое** решение.

**Ключевые тезисы и обоснования:**
1. **Почему микросервисы, а не монолит:**
   - Python (ML) и Kotlin (бизнес-логика, БД, API) — разные экосистемы. Монолит на одном из них = неоптимальный стек.
   - Независимое масштабирование ML-воркеров (тяжёлые) и API (лёгкие).
   - Изоляция отказов: упавший Python-инференс не валит REST-сервис.
2. **Почему Kafka, а не синхронный gRPC между API и воркером:**
   - Долгое время инференса (десятки секунд) — синхронный вызов держит соединение, плохо при scale.
   - Отсутствие GPU = задача в очереди ждёт. Воркер видит её сразу когда освобождается.
   - Гарантии at-least-once доставки для надёжности.
3. **Почему REST для frontend ↔ kotlin-service:** простота, отладка, совместимость с любым фронтом.
4. **Почему PostgreSQL для метаданных + MinIO для бинарников:** разделение реляционных данных (статусы задач, история, аналитика) и больших объектов (изображения, .obj/.glb).
5. **Почему отдельный frontend (React):** независимый деплой, поддержка статики через CDN.
6. **Диаграммы:** компонентная диаграмма, диаграмма последовательностей (типичный путь запроса), ER-диаграмма БД.

**Связь:** глава 4 = "что задумано", глава 5 = "как реализовано", глава 7 = "какова производительность".

**Ключевые тезисы:**

6. **Развёртывание:** docker-compose для локального dev, готовность к Kubernetes.
7. **Логирование и мониторинг:** Loki + Grafana, структурированные JSON-логи, метрики (опционально через Prometheus).

### Глава 6. Методика дообучения TripoSR

**Цель:** предложить методику улучшения качества модели для специализированных датасетов, оценить ожидаемое улучшение.

**Ключевые тезисы:**
1. **Постановка задачи дообучения:** базовая TripoSR обучена на общем корпусе; на узких категориях (например, мебель, авто) можно поднять метрики.
2. **Выбор датасета:** ShapeNet (категории chairs/cars/tables), 5—10 тысяч объектов.
3. **Препроцессинг:** разрешение 512×512, нормализация, augmentation (повороты, цвет).
4. **Процедура дообучения:**
   - lr = 1e-5 (низкий, чтобы не разрушить претрейн)
   - optimizer = AdamW
   - epochs = 5—10
   - batch size = 1—4 (ограничения по GPU)
   - заморозка ранних слоёв транспонера (только последние 2—3 блока обучаются)
5. **Метрики оценки:** Chamfer Distance, F-score, IoU.
6. **Прогнозируемые результаты:** на основе литературы по fine-tune подобных моделей — улучшение Chamfer Distance на 5-7%, F-score на 3-5%.
7. **Honest framing:** *"эмпирическая валидация — задача дальнейших исследований; в рамках данной работы предложена методика и оценка ожидаемого улучшения на основе литературных данных"*.
8. **График:** ожидаемая динамика метрик во времени обучения (loss curve, validation metric curve).

**Связь:** глава 6 показывает что платформа из глав 4-5 не просто демонстрационная — она поддерживает обновление модели.

### Глава 7. Экспериментальное исследование

**Цель:** оценить производительность реализованной платформы и подтвердить выполнение нефункциональных требований из главы 3.

**Ключевые тезисы:**
1. **Методология:** k6 как нагрузочный инструмент, сценарии "нагрузка на API" и "нагрузка на ML-воркер".
2. **Эксперимент 1: latency API.** Замер p50/p95/p99 при 100/500/1000 RPS. Целевая метрика — p95 ≤ 200 мс на легковесных эндпоинтах.
3. **Эксперимент 2: throughput ML-воркера.** Сколько 3D-моделей в минуту обрабатывает один воркер на CPU vs GPU.
4. **Эксперимент 3: масштабирование.** 1, 2, 4 воркера — линейность роста throughput, средняя задержка задачи в очереди.
5. **Эксперимент 4: сравнение с очередью и без.** Условный синхронный API (без Kafka) vs текущая асинхронная архитектура — насколько лучше выживает под пиковой нагрузкой.
6. **Анализ результатов:** соответствие требованиям из главы 3, узкие места, предложения по оптимизации.

**Honest framing:** часть экспериментов проводится на реальной системе (например, latency API на localhost), часть — методологически с прогнозируемыми числами (например, 4-кратное масштабирование GPU воркеров) если железо не позволяет провести напрямую. Числа должны быть **внутренне согласованы** (рост throughput сублинеен с ростом числа воркеров из-за overhead).

**Связь:** глава 7 закрывает цикл "требования → дизайн → реализация → подтверждение".

## Сквозные темы (упоминать по необходимости в нескольких главах)

- **Trade-offs.** Каждое решение — это компромисс. Подчёркивать (например, gRPC даёт скорость но добавляет сложность; асинхронность улучшает throughput но усложняет debugging).
- **Производственная реалистичность.** Работа описывает не учебный прототип, а систему с production-mindset: логирование, мониторинг, отказоустойчивость, контрактное API.
- **Объективная оценка.** Где использованы прогнозируемые числа — явно говорить *"оценка по литературе"*, не выдавать за измеренные.
